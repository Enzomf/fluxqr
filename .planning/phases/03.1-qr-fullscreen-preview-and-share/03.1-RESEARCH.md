# Phase 03.1: QR Fullscreen Preview & Share - Research

**Researched:** 2026-03-11
**Domain:** React dialog UI, Web Share API, CSS transform-origin animation, clipboard API
**Confidence:** HIGH

## Summary

This phase adds a fullscreen QR preview dialog triggered by clicking the 40px thumbnail in `QrListRow`. The dialog uses `@base-ui/react/dialog` (already installed at v1.2.0), surfaced via a new `dialog.tsx` shadcn component installable via `npx shadcn@latest add dialog`. The component API, data-attribute animation hooks, and Escape/backdrop-click dismiss are all built in.

The grow-from-thumbnail animation is the most technically nuanced piece. Since no Framer Motion is allowed, the approach is: capture `getBoundingClientRect()` of the thumbnail `<img>` into a ref when the user clicks, then compute `transform-origin` in viewport coordinates, and apply a custom CSS keyframe that scales from a small value to full size. The @base-ui Dialog's `data-open` / `data-closed` state attributes drive when the animation class is applied. tw-animate-css (`animate-in`, `zoom-in-*`, `animate-out`, `zoom-out-*`) works for simple scale animations but cannot encode a dynamic `transform-origin` from runtime DOM measurements — so the open/close animation for the QR image itself requires an inline `style` for `transform-origin` plus a custom `@keyframes qr-pop-open` / `qr-pop-close` added to `globals.css`.

Web Share API (`navigator.share`) is typed in `lib.dom.d.ts` and available in this project's TypeScript config. `navigator.canShare` is also typed and should be used as the guard. The `use-copy-to-clipboard` hook (new) wraps `navigator.clipboard.writeText()` with a boolean state reset on a 2-second timer.

**Primary recommendation:** Install shadcn Dialog via CLI, hand-write `QrPreviewDialog` as a controlled component receiving open state from `QrListRow`, use a `useRef` on the thumbnail `<img>` to capture position for animation origin, and implement share/copy as inline client-side logic in the dialog component.

### Pattern 1: Controlled Dialog State in QrListRow

**What:** `QrListRow` owns `open` state and the thumbnail `ref`. It passes `open`, `onOpenChange`, and `thumbnailRef` to `QrPreviewDialog`. The dialog is rendered as a sibling inside `QrListRow`'s return, not in a portal that QrList manages.

**When to use:** State is local to the row — no need to hoist to QrList. Each row has its own independent dialog instance.

**Example:**
```typescript
// src/components/dashboard/qr-list-row.tsx
'use client'

import { useState, useRef } from 'react'
import { QrPreviewDialog } from './qr-preview-dialog'

export function QrListRow({ qr, onDelete, pulseId }: QrListRowProps) {
  const [previewOpen, setPreviewOpen] = useState(false)
  const thumbnailRef = useRef<HTMLImageElement>(null)

  return (
    <QrPulseWrapper trigger={pulseId === qr.id}>
      <div className="...">
        {/* Thumbnail — clickable */}
        <img
          ref={thumbnailRef}
          src={qr.dataUrl}
          alt={qr.label}
          width={40}
          height={40}
          className="rounded shrink-0 cursor-pointer hover:opacity-80 transition-opacity"
          onClick={() => setPreviewOpen(true)}
        />
        {/* ... rest of row ... */}
      </div>
      <QrPreviewDialog
        open={previewOpen}
        onOpenChange={setPreviewOpen}
        qr={qr}
        thumbnailRef={thumbnailRef}
      />
    </QrPulseWrapper>
  )
}
```

### Pattern 3: Grow-from-Thumbnail Animation

**What:** Capture thumbnail DOMRect at click time, store in a ref. Compute `transform-origin` as viewport-relative position. Apply this as `style.transformOrigin` on the QR `<img>` inside the dialog. Use a custom CSS keyframe for the actual scale animation that @base-ui's `data-open` / `data-closed` cannot fully drive (because `transform-origin` must be dynamic).

**Approach — two-keyframe strategy:**

Option A: Override dialog popup classes entirely (simpler, recommended)
- Override `DialogContent` className to remove the default `zoom-in-95` classes
- Add custom `data-open:animate-[qr-pop-open]` + `data-closed:animate-[qr-pop-close]`
- Set `style={{ transformOrigin: computedOrigin }}` on the dialog popup element

Option B: Animate only the QR `<img>` inside the dialog, let the backdrop/overlay use defaults
- Dialog popup uses default animate-in/out (fade + zoom from center)
- Inner QR `<img>` gets its own separate scale animation from the thumbnail rect
- Backdrop uses default fade

**Recommended: Option B** — separates concerns. The dialog container fades/scales from center (good enough), and the QR image inside gets a "pop" effect via a custom `@keyframes` applied on mount. This avoids the complexity of computing transform-origin for the entire dialog popup.

**globals.css additions:**
```css
@keyframes qr-pop-open {
  from {
    opacity: 0;
    transform: scale(0.15);
  }
  to {
    opacity: 1;
    transform: scale(1);
  }
}

@keyframes qr-pop-close {
  from {
    opacity: 1;
    transform: scale(1);
  }
  to {
    opacity: 0;
    transform: scale(0.15);
  }
}
```

```typescript
// In QrPreviewDialog — dynamic transform-origin on QR image
const origin = useThumbnailOrigin(thumbnailRef, open)

<img
  src={qr.dataUrl}
  style={{ transformOrigin: origin ?? 'center' }}
  className={cn(
    'w-[280px] h-[280px] rounded',
    open ? 'animate-[qr-pop-open_300ms_ease-out_forwards]'
         : 'animate-[qr-pop-close_200ms_ease-in_forwards]'
  )}
/>
```

**Computing origin from thumbnail ref:**
```typescript
function useThumbnailOrigin(
  ref: React.RefObject<HTMLImageElement | null>,
  open: boolean
): string | null {
  const [origin, setOrigin] = useState<string | null>(null)
  useEffect(() => {
    if (open && ref.current) {
      const rect = ref.current.getBoundingClientRect()
      const cx = rect.left + rect.width / 2
      const cy = rect.top + rect.height / 2
      // transform-origin relative to viewport center (where dialog is fixed)
      const vw = window.innerWidth / 2
      const vh = window.innerHeight / 2
      setOrigin(`${cx - vw + 140}px ${cy - vh + 140}px`)
      // 140 = half of 280px QR size to convert to element-relative coords
    }
  }, [open, ref])
  return origin
}
```

This logic can live inline in `QrPreviewDialog` or extracted to a small helper.

### Pattern 4: use-copy-to-clipboard Hook

**What:** Wraps `navigator.clipboard.writeText()`. Returns `[copied, copy]` where `copied` resets to `false` after 2000ms.

```typescript
// src/hooks/use-copy-to-clipboard.ts
'use client'

import { useState, useCallback, useRef } from 'react'

export function useCopyToClipboard() {
  const [copied, setCopied] = useState(false)
  const timerRef = useRef<ReturnType<typeof setTimeout> | null>(null)

  const copy = useCallback(async (text: string) => {
    try {
      await navigator.clipboard.writeText(text)
      setCopied(true)
      if (timerRef.current) clearTimeout(timerRef.current)
      timerRef.current = setTimeout(() => setCopied(false), 2000)
    } catch {
      // silently fail — clipboard may be denied
    }
  }, [])

  return { copied, copy }
}
```

### Pattern 5: Web Share API Guard

**What:** `navigator.share` is only available on mobile browsers and some desktop Chrome/Edge. Must guard at runtime, not at render time.

```typescript
// In QrPreviewDialog — correct TypeScript pattern
const canShare = typeof navigator !== 'undefined' && 'share' in navigator

// Share handler
async function handleShare() {
  if (!canShare) return
  await navigator.share({
    title: qr.label,
    url: `${process.env.NEXT_PUBLIC_SITE_URL}/q/${qr.slug}`,
  })
}
```

The `navigator.share` TypeScript type is `(data?: ShareData) => Promise<void>` — already in `lib.dom.d.ts`. No type casting needed.

### Anti-Patterns to Avoid

- **Hoisting open state to QrList:** Creates a single dialog for all rows with complex "which QR is open" tracking. One dialog per row is simpler and correct.
- **Using DialogTrigger wrapping the thumbnail:** Loses the ability to capture the ref AND conflicts with the existing `<img>` element. Use controlled `open`/`onOpenChange` instead.
- **Calling `navigator.share` without a guard:** Throws on desktop Firefox/Safari. Always check `'share' in navigator` first.
- **Computing transform-origin once at component mount:** The thumbnail position changes if the user scrolls or resizes before opening. Capture the rect at click time (in the onClick handler), not in a one-time effect.
- **Applying `animate-[qr-pop-close]` via CSS class alone:** @base-ui removes the popup from the DOM when `onOpenChangeComplete` fires. To see the close animation, the popup must stay mounted during the animation. Use `onOpenChangeComplete` (fires after animation) for cleanup, or rely on @base-ui's built-in mounting behavior which keeps the element until transition finishes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Modal focus trap | Custom focus management | @base-ui Dialog (built-in focus trap, aria-modal) | Focus trap has many edge cases |
| Escape key handling | keydown listener | @base-ui Dialog (built-in) | Already wired to `onOpenChange(false)` |
| Backdrop click dismiss | Manual click handler on overlay | @base-ui Dialog (built-in, `disablePointerDismissal=false`) | Race conditions with propagation |
| Scroll lock | Body overflow manipulation | @base-ui Dialog (built-in `modal=true`) | Handles scrollbar width compensation |
| Clipboard API fallback | `document.execCommand('copy')` | `navigator.clipboard.writeText()` | `execCommand` deprecated; project targets modern browsers |

**Key insight:** @base-ui Dialog handles all the accessibility primitives that look simple but have many edge cases (nested dialogs, scroll restoration, focus return on close). Never replace these with custom solutions.

## Common Pitfalls

### Pitfall 1: Transform-Origin Coordinates in a Fixed-Position Dialog

**What goes wrong:** The QR preview dialog popup is `position: fixed; top: 50%; left: 50%; transform: translate(-50%, -50%)`. If you set `transform-origin` using raw viewport coordinates from `getBoundingClientRect()`, the origin is in viewport space but the CSS transform is applied relative to the element's own bounding box. The calculation must convert viewport-space thumbnail center to element-relative coordinates.

**Why it happens:** Mixing viewport coordinate system (getBoundingClientRect) with CSS transform-origin coordinate system (relative to element's transformed position).

**How to avoid:** The element center in viewport is `(left + right) / 2, (top + bottom) / 2`. For a centered fixed element of size W×H, its top-left in viewport is `(vw/2 - W/2, vh/2 - H/2)`. So element-relative origin = `(thumbnailCenterX - (vw/2 - W/2), thumbnailCenterY - (vh/2 - H/2))`.

**Warning signs:** Animation starts from wrong position, looks like it's scaling from the wrong corner.

### Pitfall 2: @base-ui Controlled vs Uncontrolled Dialog

**What goes wrong:** Passing `open` prop without `onOpenChange` — the dialog becomes read-only controlled and backdrop/Escape clicks have no effect.

**Why it happens:** @base-ui requires both `open` AND `onOpenChange` for controlled mode.

**How to avoid:** Always pass both: `<Dialog open={open} onOpenChange={setOpen}>`.

### Pitfall 3: Close Animation Cutoff

**What goes wrong:** When the user closes the dialog, the close animation plays but the popup disappears immediately before the animation completes.

**Why it happens:** @base-ui keeps the element mounted during the exit transition. The default `data-closed:animate-out` classes work with `duration-100` to complete before unmount. Custom longer close animations (200-300ms) need matching `duration-200`/`duration-300` on the popup.

**How to avoid:** Add `duration-300` (or the actual animation duration) to the `DialogPrimitive.Popup` className so @base-ui waits that long before unmounting.

**Warning signs:** Dialog disappears with no animation on close, or animation cuts off halfway.

### Pitfall 4: `navigator.share` Promise Rejection on User Cancel

**What goes wrong:** On mobile, if the user opens the share sheet and then cancels, `navigator.share()` rejects with `AbortError`. Unhandled rejection causes console errors.

**Why it happens:** Canceling the native share sheet is a user-initiated abort, which the browser surfaces as a rejected promise.

**How to avoid:**
```typescript
try {
  await navigator.share({ title: qr.label, url: shareUrl })
} catch (err) {
  if (err instanceof Error && err.name !== 'AbortError') {
    // only log unexpected errors
    console.error(err)
  }
}
```

### Pitfall 5: `asChild` Not Supported on @base-ui DialogTrigger

**What goes wrong:** Wrapping the `<img>` thumbnail in `<DialogTrigger asChild>` to make it the trigger — @base-ui's Trigger does not support `asChild` (unlike Radix).

**Why it happens:** @base-ui uses a `render` prop pattern instead of `asChild`. DialogTrigger renders its own `<button>` by default.

**How to avoid:** Use controlled mode (`open` + `onOpenChange` on Dialog.Root) and an `onClick` on the `<img>` directly. This is the established pattern in the project (see AlertDialog usage in `delete-button.tsx`).

## Open Questions

1. **Transform-origin accuracy on scroll**
   - What we know: `getBoundingClientRect()` returns viewport-relative coordinates, accounting for scroll
   - What's unclear: If the user has scrolled the dashboard list before clicking, the calculation should still be correct since getBoundingClientRect is always viewport-relative
   - Recommendation: Capture the rect in the `onClick` handler (not in a useEffect) to ensure it's fresh at click time

2. **Close animation with @base-ui unmounting timing**
   - What we know: @base-ui keeps the popup mounted during the CSS transition duration specified by `duration-*` class on the popup
   - What's unclear: Whether `duration-300` on the popup element is sufficient to hold the element for the close keyframe
   - Recommendation: Test empirically; if the dialog disappears before close animation completes, add `onOpenChangeComplete` to reset state after animation

3. **`NEXT_PUBLIC_SITE_URL` in client component**
   - What we know: It's a `NEXT_PUBLIC_` prefixed env var — safe on client
   - What's unclear: Whether it's reliably set in all environments (dev/prod)
   - Recommendation: Default to `window.location.origin` as fallback: `` const siteUrl = process.env.NEXT_PUBLIC_SITE_URL ?? (typeof window !== 'undefined' ? window.location.origin : '') ``
