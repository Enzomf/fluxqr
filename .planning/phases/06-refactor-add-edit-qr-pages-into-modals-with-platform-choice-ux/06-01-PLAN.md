**Part B — Modify `src/components/qr-management/qr-form.tsx`:**

Extend `QrFormProps` interface:
```typescript
interface QrFormProps {
  action: (prevState: FormState, formData: FormData) => Promise<FormState>
  defaultValues?: Partial<QrCode>
  mode: 'create' | 'edit'
  verifiedPhone?: string | null
  qrType?: 'default' | 'custom'  // NEW — controls message field visibility
  onSuccess?: (id?: string) => void  // NEW — called on successful save
}
```

Changes to QrForm:
1. Change the `FormState` import from `@/app/dashboard/new/actions` to `@/app/dashboard/qr-actions`.
2. Add `qrType` and `onSuccess` to destructured props (defaults: `qrType = 'custom'`, `onSuccess = undefined`).
3. Add a `useEffect` watching `state.success`: when `state.success` is true, call `router.refresh()`, then `onSuccess?.(state.id)`, then show toast (`toast.success(mode === 'create' ? 'QR criado com sucesso' : 'QR atualizado')`). Import `toast` from `sonner`.
4. Conditionally render the "Default Message" textarea block: only show it when `qrType === 'custom'`. When `qrType === 'default'`, do NOT render the textarea and do NOT include a hidden input for `default_message` (the server action already defaults it to `''`).
5. Remove the `max-w-lg` class from the `<form>` element — the dialog will control width. Keep `space-y-6`.
6. Add `id="qr-form"` to the `<form>` element so the sticky footer submit button (in the dialog shell, Plan 02) can reference it via `form="qr-form"`.
7. Remove the submit `<Button>` from inside the form — it will be rendered in the dialog's sticky footer instead. The form now ends after the last field.

Keep all existing functionality: phone verification banner, justVerified state, validation error display, SlugInput, PlatformSelector, contact target read-only chip.
