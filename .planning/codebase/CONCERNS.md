## Tech Debt

**Default scaffolding not replaced:**
- Issue: `app/page.tsx` is the unmodified `create-next-app` default — displays Next.js logo, Vercel deploy links, and "edit page.tsx" copy. This will be served at the root `/` route in production until replaced.
- Files: `app/page.tsx`
- Impact: Root URL renders a framework boilerplate page instead of the login or redirect flow.
- Fix approach: Replace with the login redirect (`/login`) or a marketing landing page as the first implementation task.

**Root layout uses wrong font loading strategy:**
- Issue: `app/layout.tsx` loads Geist via `next/font/google` (network fetch), but `CLAUDE.md` specifies Geist via the `geist` npm package (`GeistSans` / `GeistMono` imports) which is already present as a transitive dependency of Next.js 16.
- Files: `app/layout.tsx`
- Impact: Font may load from Google Fonts CDN in production, adding an external network dependency and a potential FOUT on slow connections. The `geist` package enables fully self-hosted fonts.
- Fix approach: Switch to `import { GeistSans } from "geist/font/sans"` and `import { GeistMono } from "geist/font/mono"` per the BACKLOG.MD Task 00 spec.

**`globals.css` uses light-mode defaults and system-preference media query:**
- Issue: `app/globals.css` sets `--background: #ffffff` and uses `@media (prefers-color-scheme: dark)` to conditionally apply dark styles. The CLAUDE.md architecture locks the app to dark mode permanently via `class="dark"` on `<html>`. The current CSS will conflict once `class="dark"` is added.
- Files: `app/globals.css`
- Impact: Until `globals.css` is replaced, any component using `--background` or `--foreground` CSS variables will render with wrong colors (white background instead of `#0F172A`).
- Fix approach: Replace `globals.css` content with the shadcn CSS variable overrides specified in BACKLOG.MD Task 00, step 5.

**`app/layout.tsx` metadata is placeholder copy:**
- Issue: `metadata.title` is "Create Next App" and `metadata.description` is "Generated by create next app".
- Files: `app/layout.tsx`
- Impact: All pages share this boilerplate title/description in browser tabs, search results, and social previews.
- Fix approach: Update to "FluxQR" / "Scan to connect — WhatsApp, SMS, Telegram" before first deploy.

**`tsconfig.json` path alias maps `@/*` to `./` (project root), not `./src/`:**
- Issue: The alias `"@/*": ["./*"]` resolves from the project root. The planned directory structure (per `CLAUDE.md`) places all source files under `src/`. If `src/` is created, imports like `@/components/ui/button` will need the alias updated to `"@/*": ["./src/*"]`, or the directory structure must be kept flat (no `src/` wrapper).
- Files: `tsconfig.json`
- Impact: All imports will silently resolve to the wrong path if `src/` is introduced without updating the alias. TypeScript will error or resolve files unexpectedly.
- Fix approach: Decide on directory structure before writing any application code. If using `src/`, update `tsconfig.json` paths to `"@/*": ["./src/*"]`. If keeping flat `app/` at root, update `CLAUDE.md` to reflect actual structure.

**Version mismatch between CLAUDE.md spec and installed packages:**
- Issue: `CLAUDE.md` specifies "Next.js 15" but `package.json` has `"next": "16.1.6"`. Similarly, the stack says "React 19" but `package.json` has `"react": "19.2.3"`.
- Files: `package.json`, `CLAUDE.md`
- Impact: Low for React (19.2.3 is a React 19 patch). For Next.js, the difference matters: APIs, middleware behavior, and some App Router internals changed between 15 and 16. Any implementation following BACKLOG.MD examples (written for Next.js 15) could encounter API differences.
- Fix approach: Update `CLAUDE.md` to reflect Next.js 16.x. Review BACKLOG.MD task implementations for any Next.js 15-specific patterns before coding.

## Security Considerations

**No middleware protecting `/dashboard/*`:**
- Risk: Once auth is implemented, unauthenticated users can access dashboard routes if middleware is not in place.
- Files: `middleware.ts` — does not yet exist
- Current mitigation: None. Application has no auth flows implemented yet.
- Recommendations: Implement middleware in Task 02 before any dashboard routes are built. Gate the entire `/dashboard/:path*` prefix.

**`SUPABASE_SERVICE_ROLE_KEY` has no guard in place:**
- Risk: The service role key bypasses all RLS. If accidentally exposed via a `NEXT_PUBLIC_` prefix or leaked through a client component, it grants full database access.
- Files: Not yet implemented — risk is architectural
- Current mitigation: No Supabase code exists yet, so no exposure yet.
- Recommendations: Never prefix with `NEXT_PUBLIC_`. Only import in server-side files (`lib/supabase/server.ts`). Add a build-time check or ESLint rule to reject `NEXT_PUBLIC_SERVICE_ROLE` pattern.

**`/api/slug-check` endpoint is unauthenticated by design but could be abused:**
- Risk: The planned `GET /api/slug-check?slug=` route returns availability for any slug without auth. This exposes the full namespace of taken slugs via enumeration.
- Files: `app/api/slug-check/route.ts` — not yet implemented
- Current mitigation: None planned in BACKLOG.MD.
- Recommendations: Add rate limiting (e.g., Vercel Edge rate limit headers or a simple in-memory counter). Consider returning only boolean without leaking timing information.

## Performance Bottlenecks

**Scanner page 10KB JS budget has no enforcement mechanism:**
- Problem: BACKLOG.MD and `CLAUDE.md` require `/q/[slug]` to ship under 10KB of JS. There is currently no bundle analyzer or CI check enforcing this.
- Files: `app/q/[slug]/` — not yet implemented, `next.config.ts`
- Cause: No `bundleAnalyzer` config, no size-limit check in CI.
- Improvement path: Add `@next/bundle-analyzer` to `next.config.ts` and/or a `size-limit` package to `package.json` with a CI check targeting the scanner route chunk.

**QR image generation is client-side only:**
- Problem: The planned `generateQrDataUrl()` runs in the browser via `use-qr-image.ts` hook. Every dashboard list row generates its QR data URL client-side on mount, causing a burst of canvas operations.
- Files: `lib/qr-generator.ts`, `hooks/use-qr-image.ts`, `components/qr-generation/qr-image.tsx` — not yet implemented
- Cause: `qrcode` package runs in the browser; data URLs are regenerated on every page load.
- Improvement path: Generate QR data URLs server-side and cache them, or generate once and store the URL. For MVP, ensure `use-qr-image.ts` uses `useMemo` and only regenerates when `slug` changes.

## Fragile Areas

**`app/q/[slug]/page.tsx` fire-and-forget RPC:**
- Files: `app/q/[slug]/page.tsx` — not yet implemented
- Why fragile: BACKLOG.MD specifies `increment_scan_count` as "fire-and-forget, non-blocking." In Next.js App Router Server Components, `await` is required to complete Supabase calls before the response is sent. Omitting `await` on the RPC means the increment may never execute if the serverless function terminates before the promise resolves.
- Safe modification: Either `await` the RPC and accept the minor latency, or use `waitUntil()` from the Vercel runtime if available in Next.js 16. Document the tradeoff explicitly.
- Test coverage: None yet.

**Soft delete visibility gap:**
- Files: `app/q/[slug]/page.tsx`, `app/dashboard/page.tsx` — not yet implemented
- Why fragile: The proxy route queries `WHERE is_active = true`. The dashboard queries `WHERE is_active = true`. A soft-deleted row (`is_active = false`) disappears from both views but remains in the database. If `revalidatePath` is not called correctly after delete, stale cache can show the row as still active in the dashboard.
- Safe modification: Always call `revalidatePath('/dashboard')` after any `is_active` update. Add a Supabase RLS SELECT policy that also respects `is_active` for owner queries to prevent confusion.
- Test coverage: None yet.

## Test Coverage Gaps

**No test infrastructure exists:**
- What's not tested: Everything. No test runner, no test files, no test utilities.
- Files: No `*.test.*` or `*.spec.*` files exist. No `jest.config.*`, `vitest.config.*`, or testing dependencies in `package.json`.
- Risk: All future code ships without automated verification. Regression detection relies entirely on manual testing.
- Priority: High — especially for `lib/redirect.ts` (deep link URL construction), `lib/qr-generator.ts`, and Supabase server actions which have security implications.

**`buildPlatformUrl()` has no unit tests:**
- What's not tested: The `redirect.ts` function that constructs `wa.me`, `sms:`, and `t.me` deep links. URL encoding errors or phone number stripping bugs would silently send users to broken links.
- Files: `lib/redirect.ts` — not yet implemented
- Risk: Incorrect deep links break the core product flow (scan → open messaging app). Bugs are only detectable via manual device testing.
- Priority: High — add unit tests covering all three platforms, special characters in messages, and edge cases (international phone numbers, empty messages).

**RLS policies have no integration tests:**
- What's not tested: The Supabase RLS policies ensuring users can only access their own `qr_codes` rows.
- Files: Supabase migration files — not yet created
- Risk: An RLS misconfiguration could expose one user's QR codes to another, or allow unauthenticated users to modify records.
- Priority: High — use Supabase's `pgTAP` or a local Supabase integration test suite to verify all 5 RLS policies.

## Scaling Limits

**`scan_count` is a single integer column with no time-series data:**
- Current capacity: Unlimited total count via atomic RPC increment.
- Limit: The schema has no `scan_events` table. Analytics beyond total count (daily, weekly trends, geo, device) cannot be added without a schema migration.
- Scaling path: BACKLOG.MD Task 10 notes to "reserve space for a future sparkline chart (v1.1 `scan_events` table)." Plan the `scan_events` schema before v1.1 to avoid a breaking migration.

**Single Supabase project for all environments:**
- Current capacity: Unknown — depends on Supabase plan.
- Limit: If development, staging, and production share a single Supabase project, a schema migration error in dev can corrupt production data.
- Scaling path: Create separate Supabase projects per environment. Use `supabase link` to manage project targeting.

## Dependencies at Risk

**`next: 16.1.6` is not a stable release channel version:**
- Risk: Next.js 16 is not listed on the official Next.js stable releases page as of this analysis date. This version may be a release candidate or early access build. It may contain breaking changes, unfixed bugs, or undocumented API differences from Next.js 15.
- Impact: Middleware behavior, Server Actions API, App Router caching, and `@supabase/ssr` compatibility are all potentially affected.
- Migration plan: Verify that `16.1.6` is intentional. If not, pin to `15.x` which is documented stable. If intentional, closely track Next.js 16 changelog for breaking changes.
